# 09 AI 功能方案（作品集版）

> 使用 skills：`prd-development`、`lean-ux-canvas`。当前 AI 演示使用本地 mock 输出，不接真实模型 API。

## AI 定位

邻宠到家的 AI 不是泛聊天入口，而是嵌入宠物照护业务链路的辅助能力：

- 预约前：回答标准服务问题，降低信息不确定性。
- 服务中/后：根据服务记录生成日报草稿，提升过程透明。
- 运营侧：生成多渠道文案，提高内容生产效率。
- 复盘侧：把模拟经营数据解释为“洞察 - 证据 - 建议”。

AI 负责信息整理、内容草稿和决策建议；涉及医疗、投诉、赔付、异常履约和真实承诺时必须人工确认。

## 功能总览

| 模块 | 用户 | 输入 | 输出 | 边界 |
| --- | --- | --- | --- | --- |
| AI 客服助手 | 宠物主人、客服 | 用户问题、服务类型、宠物信息、服务规则 | 直接回答、补充信息、下一步建议、风险提示 | 不做医疗诊断，不承诺赔付 |
| AI 寄养日报 | 服务者、宠物主人 | 订单、服务记录、异常备注 | 今日状态、完成事项、照片说明、关注事项、建议 | 不编造服务动作，不自动发送 |
| AI 运营文案 | 运营人员 | 渠道、服务类型、目标用户、卖点、语气 | 标题、正文、卖点、CTA、合规提醒 | 不编造真实评价，不夸大安全 |
| AI 经营分析 | 运营人员、面试官 | 指标卡、趋势、渠道、服务结构 | 优先级、洞察、证据、建议、人工确认点 | 不把模拟数据当真实结论 |

## Prompt 结构原则

每个 Prompt 都包含：

- 角色：说明 AI 是客服、日报助手、文案助手还是经营分析助手。
- 业务上下文：服务类型、宠物信息、规则、指标或渠道。
- 输出结构：固定字段，便于前端展示和人工审核。
- 禁止事项：医疗诊断、赔付承诺、编造事实、自动执行。
- 人工确认：高风险问题转人工，内容发布前人工确认。

## 风险边界

- 不做宠物医疗诊断或急救判断。
- 不处理退款、投诉、赔付和责任归因。
- 不自动发布日报、文案或经营策略。
- 不泄露用户或照护者隐私。
- 不把 mock AI 输出包装成真实模型能力。

## 评估指标

| 模块 | 指标 |
| --- | --- |
| AI 客服 | 首次回答解决率、转人工率、咨询到预约转化率、满意度 |
| AI 日报 | 草稿采纳率、编辑比例、日报查看率、异常漏报率 |
| AI 文案 | 草稿采纳率、人工编辑幅度、渠道点击率、预约转化率 |
| AI 分析 | 洞察采纳率、建议转化为动作比例、复盘耗时下降、证据引用准确率 |

## 当前实现方式

- 在前端使用 mock 数据和 mock AI engine。
- 用短暂 loading 模拟生成过程。
- 在页面标注 mock AI 输出。
- 所有高风险场景展示人工确认或风险提示。

## 后续真实接入方案

若未来接真实模型，应采用 Node 后端中转：

- API Key 只放后端环境变量。
- 后端负责拼接 Prompt、调用模型、日志记录和频控。
- 对输入输出做脱敏和高风险关键词拦截。
- 前端只展示结果，不直接调用模型。
- 重要输出进入人工确认状态。
