# 06 PRD - 邻宠到家（作品集版）

> 使用 skills：`prd-development`、`problem-statement`、`proto-persona`、`user-story-mapping`。本 PRD 面向求职作品集展示，不代表真实上线需求。

## 1. 产品概述

邻宠到家是一个社区宠物照护预约 MVP，面向需要临时安排宠物照护的城市宠物主人，提供上门遛狗、上门喂猫、宠物日托和服务追踪。当前版本使用本地 mock 数据和微信 storage，不接后端、不接云开发、不接真实支付。

## 2. 问题与机会

宠物主人在加班、出差、旅行或临时外出时，需要快速找到可信照护者；但现有熟人托付、微信群和线下门店预约存在信息不标准、过程不透明、风险边界不清的问题。邻宠到家通过结构化预约、照护者卡片、订单记录、服务反馈和 AI 演示模块展示一套可落地的产品方案。

## 3. 目标用户

- **宠物主人**：需要快速安排上门喂猫、遛狗或日托，关注可信度、价格、服务过程。
- **照护者**：提供附近宠物照护服务，关注订单信息清晰、服务记录标准化。
- **运营人员**：关注渠道转化、服务结构、订单确认、履约、复购和异常风险。
- **面试官**：关注候选人是否能从业务问题推导到产品方案、指标和边界。

## 4. MVP 范围

### 用户端

- 首页展示品牌、服务、流程、附近照护者、价格和 CTA。
- 预约页支持服务类型、地址、日期、时间段、宠物信息、备注。
- 价格根据服务类型、晚间时段和关键词进行本地预估。
- 订单页读取本地 storage，按最新订单排序，支持清空。
- 订单详情展示服务、宠物、地址、时间、价格和状态。

### 照护者端展示

- 照护者列表支持按服务类型筛选。
- 照护者详情展示片区、评分、服务次数、擅长类型和服务说明。

### 运营与 AI 展示

- 看板展示模拟预约、确认订单、收入、入住率、复购率、渠道和漏斗。
- AI 功能演示客服、日报、运营文案和经营分析，全部使用 mock 输出。

## 5. 非目标范围

- 不做真实注册登录、真实支付、真实接单、真实地图调度、真实客服 IM。
- 不处理真实退款、投诉、赔付、宠物医疗诊断。
- 不接真实 AI API，不保存真实用户个人信息。
- 不承诺真实商业增长结果。

## 6. 关键流程

1. 用户进入首页，理解服务和信任机制。
2. 用户选择服务类型，进入预约页。
3. 用户填写地址、日期、时间段、宠物信息和备注。
4. 系统本地计算预估价格。
5. 用户提交预约，本地生成订单并保存到 storage。
6. 用户在订单页查看状态和详情。
7. 运营侧通过看板查看模拟数据，并通过 AI 经营分析形成建议。

## 7. 数据与类型

小程序 MVP 使用：

- `ServiceType`: `dogWalking` / `catFeeding` / `dayCare`
- `OrderStatus`: `pending` / `serving` / `completed`
- `BookingForm`: 服务类型、地址、日期、时间段、宠物名、宠物类型、宠物备注、用户备注。
- `Order`: 继承预约字段，增加 id、创建时间、预估价格、状态。

## 8. 价格规则

- 上门遛狗基础价：39。
- 上门喂猫基础价：49。
- 宠物日托基础价：129。
- 备注包含“多宠物”“多猫”“节假日”“药”时增加 20。
- 时间段为 `18:00 - 20:00` 时增加 10。

## 9. 成功指标

| 指标 | 口径 | 作品集价值 |
| --- | --- | --- |
| 预约提交率 | 提交预约 / 进入预约页 | 衡量表单是否清晰 |
| 订单确认率 | 确认订单 / 提交预约 | 衡量信任与价格接受度 |
| 日报查看率 | 查看日报 / 完成服务 | 衡量过程反馈价值 |
| 复购率 | 复购用户 / 完成服务用户 | 衡量服务满意和信任沉淀 |
| 渠道确认转化率 | 确认订单 / 渠道访问 | 衡量运营渠道质量 |

## 10. 验收标准

- 面试官能从 README 进入 PRD、小程序、看板、AI 方案和视频。
- 小程序 MVP 能在微信开发者工具中展示完整用户路径。
- 看板能解释模拟经营数据的业务含义。
- AI 方案包含输入、输出、人工确认、风险边界和评估指标。
- 所有文档明确标注模拟数据和 mock AI 边界。

## 11. 风险与约束

- 真实宠物照护涉及安全、隐私、责任和线下履约，作品集只展示信息架构和原型。
- AI 不能做医疗判断、赔付承诺、自动发布日报或自动调整运营策略。
- 模拟数据只能用于展示指标设计，不能作为真实市场结论。
